# 久坐检测调试指南

## 问题现象
- 久坐事件发送后，日志只显示"清除久坐状态"
- 主页没有黄色氛围光
- 记录页没有久坐记录

## 调试步骤

### 1. 查看详细日志
运行应用后，查找以下关键日志：

```
[mqtt]:📨 handleMessage 开始处理 topic=...
[mqtt]:收到原始消息 topic=..., 内容=...
[mqtt]:解析JSON成功, event=..., alert_type=..., type=...
[mqtt]:事件字段值="..."
[mqtt]:关键词检测 isEmpty=..., hasFall=..., hasSedentary=..., hasNormal=...
[mqtt]:事件检测结果 isFall=..., isSedentary=...
```

**关键点：**
- `hasSedentary` 应该是 `true`
- `hasNormal` 应该是 `false`
- `isSedentary` 应该是 `true`

### 2. 发送测试消息

使用Python脚本发送：
```bash
python test_sedentary_debug.py
```

或使用MQTT工具（MQTTX、mosquitto_pub）：
```bash
# 发送到主题: ai_guardian/alerts/sedentary
{"event": "sedentary", "confidence": 0.95}
```

### 3. 检查日志输出

#### 如果看到 `🟡🟡🟡 检测到久坐事件`
✅ 检测正常，继续查看后续日志

#### 如果看到 `🟢 设置为正常状态` 或 `🟢 纯文本匹配：清除所有状态`
❌ 事件被误判为正常状态，检查：
- `hasNormal` 为什么是 `true`？
- 事件字段中是否包含了"normal"、"正常"等关键词？

#### 如果久坐检测为 true，但之后立即清除
检查是否有其他消息立即跟随，导致状态被清除

### 4. 常见问题

#### 问题1：事件字段为空
日志显示：`isEmpty=true`

**解决方法：**
- 检查MQTT消息格式，确保包含 `event`、`alert_type` 或 `type` 字段
- 示例：`{"event": "sedentary"}`

#### 问题2：包含正常关键词
日志显示：`hasNormal=true`

**解决方法：**
- 检查事件字段是否包含：`normal`, `正常`, `stand`, `站立`, `walk`, `行走`
- 这些关键词会导致事件被判定为正常状态

#### 问题3：久坐立即被清除
日志显示久坐检测成功，但随后看到 `清除久坐状态`

**可能原因：**
- 自动恢复定时器（3秒后自动恢复）
- 收到了后续的正常状态消息
- 收到了摔倒消息（摔倒会清除久坐状态）

### 6. UI显示检查

当 `isSedentaryDetected=true` 后，应该看到：

1. **主页 (mainpage.ets)**
   - 黄色边框氛围光（8px宽）
   - 消息区黄色背景
   - 显示：`⚠️ 提醒: 检测到久坐! @ 时间`

2. **记录页 (record.ets)**
   - 切换到"久坐记录"标签页
   - 显示久坐记录列表
   - 总久坐次数增加

3. **3秒后自动恢复**
   - 氛围光消失
   - 背景恢复正常
   - 但记录仍然保留

## 下一步操作

如果日志正常但UI不显示，检查：
1. mainpage.ets 中的 `syncState()` 是否正常同步
2. `isSedentaryDetected` 状态是否正确传递
3. 氛围光组件的条件判断 `if (this.isFallDetected || this.isSedentaryDetected)`
