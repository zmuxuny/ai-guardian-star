# MQTT 摔倒记录诊断指南

## 问题现象
- 测试脚本成功发送了 5 条 MQTT 消息
- 但应用的记录页面没有显示任何记录

## 诊断步骤

### 1. 检查 MQTT 连接状态
在应用的日志中搜索以下关键字：

```
[mqtt]:正在尝试连接 MQTT Broker...
[mqtt]:MQTT 连接成功。
[mqtt]:✅ 成功订阅主题 ai_guardian/alerts/#
[mqtt]:📡 等待接收消息...
```

**预期结果**：应该看到连接成功和订阅成功的日志

---

### 2. 检查消息接收
搜索以下日志：

```
[mqtt]:📬 messageArrived 回调触发!
[mqtt]:📨 handleMessage 开始处理
[mqtt]:收到原始消息
```

**预期结果**：
- 如果看到这些日志，说明消息已接收
- 如果没有，说明消息未到达应用（网络/订阅问题）

---

### 3. 检查摔倒检测逻辑
搜索以下日志：

```
[mqtt]:解析JSON成功
[mqtt]:事件字段值=
[mqtt]:摔倒检测结果 isFall=
[mqtt]:🔴 检测到摔倒事件!
```

**预期结果**：应该看到 `isFall=true` 和检测到摔倒的日志

---

### 4. 检查记录添加
搜索以下日志：

```
[mqtt]:📝 addFallRecord 被调用
[mqtt]:✅ 新摔倒事件已记录!
[mqtt]:数组长度变化:
[mqtt]:当前所有记录(X条):
```

**预期结果**：
- 应该看到记录被添加的日志
- 数组长度应该从 0 增加到 1, 2, 3...
- 应该能看到完整的记录列表

---

### 5. 检查 UI 更新
搜索以下日志：

```
[record]:页面加载，当前记录数=
[record]:当前记录:
```

**预期结果**：
- 切换到记录页面时应该看到正确的记录数
- 如果记录数 > 0 但看不到列表，可能是 UI 渲染问题

---

## 常见问题排查

### 问题 A: 没有 "messageArrived 回调触发" 日志
**原因**：消息未到达应用
**解决方案**：
1. 确认应用已启动并连接到 MQTT
2. 确认测试脚本和应用连接的是同一个 Broker
3. 检查防火墙设置
4. 重新运行测试脚本

### 问题 B: 有消息到达但 isFall=false
**原因**：摔倒检测逻辑判断为非摔倒
**解决方案**：
1. 查看 `事件字段值=` 后面的值是什么
2. 确认消息格式是否正确（应该包含 `"event": "fall"`）
3. 检查是否有否定关键词（如 "normal"）

### 问题 C: 记录已添加但 UI 不显示
**原因**：响应式更新或渲染问题
**解决方案**：
1. 确认 `fallRecords` 数组确实在增长
2. 尝试切换到其他 Tab 再切回记录页面
3. 检查 ForEach 的 keyGenerator 是否正确

### 问题 D: 只显示一条记录
**原因**：2秒防重复机制或数组更新问题
**解决方案**：
1. 查看 "忽略重复摔倒事件" 日志
2. 确认测试消息之间有足够的时间间隔（>2秒）
3. 检查数组是否真的在增长
